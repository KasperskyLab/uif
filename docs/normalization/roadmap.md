# Roadmap: Normalization Tool

---

## Цель

Редактор форм оперирует **ровно двумя** категориями элементов:

- **Компоненты дизайн-системы** — всё, что экспортируется из `@kaspersky/hexa-ui`.
- **Кастомные компоненты** — пользовательские элементы, подключаемые отдельно.

Других источников нет. Любая доработка ниже валидируется этим ограничением: если фича требует третью категорию или обход ДС — она вне скоупа.

---

## 1. Хендлеры компонентов

Хендлеры — **кастомные React-хуки** (модули по ссылке из формы), а не произвольные функции. Один хук получает **два независимых набора пропсов** — из редактора и из Hexa UI.

### Пропсы из инструмента (KUNT)

*Kaspersky UI Normalization Tool* — контекст формы и редактора:

- конфиг формы (`name`, `id`, `schema`, …);
- стейт формы (значения полей);
- идентификаторы и метаданные контрола в DSL, нужные инструменту.

### Пропсы из Hexa UI

То, что отдаёт компонент дизайн-системы при вызове колбэка:

- аргументы события (как у нативного `onClick` / `onChange` и т.д.);
- актуальные пропсы экземпляра компонента Hexa, если это нужно контракту хука.

Два блока явно разделены в API хука (например два аргумента или два поля объекта), чтобы не смешивать слой инструмента и слой ДС.

## 2. Хендлер настройки компонента Hexa UI

У **каждого** компонента дизайн-системы в форме — отдельный хук (модуль), задаваемый в настройках **KUNT**: это **хендлер конфигурации** экземпляра Hexa-компонента, а не обработчик события.

**Вход:**

1. **Один параметр** — вместе настройки и стейт формы (конфиг + значения полей и всё, что нужно для вывода пропсов).
2. **Второй параметр** — тип/идентификатор компонента Hexa UI (какой именно компонент ДС настраивается).

**Выход:** объект (перечень) пропсов **конкретного** экземпляра компонента дизайн-системы. Этот результат **transpile viewer** подставляет при рендере вместо статических значений из DSL, чтобы отображение совпадало с логикой, заданной в инструменте.

**Transpile viewer:** хендлер **перезапускается** при изменении **конфига** формы или **стейта** формы. Пересчёт пропсов **мемоизируется**: повторный вызов только если изменились те поля конфига и стейта, от которых зависит хендлер (явный список зависимостей в настройках или механизм уровня `useMemo` по фактически читаемым полям).

## 3. Превью в редакторе

Превью формы работает на **той же** библиотеке Hexa UI, что и холст редактора. Не должно быть двух разных рендереров / двух библиотек — одна реализация для проектировщика и для превью.

## 4. Полная поддержка компонентов ДС

- **Все экспортируемые компоненты** из `@kaspersky/hexa-ui` (index) доступны в палитре. Список не хардкодится — формируется из экспортов пакета.
- **Все пропсы** каждого компонента доступны для настройки. Список свойств и подсказки типов берутся из TypeScript-описаний (`.d.ts`), а не из ручного перечня.

## 5. Адаптация к Storybook

После стабилизации редактора — интеграция со Storybook: формы или отдельные компоненты из DSL должны отображаться как стори.
