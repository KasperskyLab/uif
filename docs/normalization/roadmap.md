# Roadmap: Normalization Tool

---

## Цель

Редактор форм оперирует **ровно двумя** категориями элементов:

- **Компоненты дизайн-системы** — всё, что экспортируется из `@kaspersky/hexa-ui`.
- **Кастомные компоненты** — пользовательские элементы, подключаемые отдельно.

Других источников нет. Любая доработка ниже валидируется этим ограничением: если фича требует третью категорию или обход ДС — она вне скоупа.

**Паритет инструментов:** каждый этап плана (пункты 1–5) выполняется **сразу для обоих** приложений — **Kaspersky UI Normalization Tool** и **transpile viewer**. Одна фича не считается закрытой, пока не согласованы редактор и просмотрщик (общий DSL, одинаковая семантика, без отставания одного пакета).

---

## 1. Конфиг-хуки экземпляров компонентов ДС

**Скоуп:** контролы **Button**, **Text**, **Grid**, **Table** — на каждом из них тот же контракт `configHook` и паритет **normalization tool** / **transpile viewer**, что и для кнопки (с типами пропсов соответствующих компонентов Hexa UI).

**Прогресс по скоупу п.1**

**Сделано**

- ✅ **Button** — [план и детали](./plan-button.md).

**В очереди**

- ⏳ **Text** — тот же контракт `configHook`, типы пропсов Hexa для `Text`.
- ⏳ **Grid** — то же для `Grid` / ячеек по правилам DSL.
- ⏳ **Table** — то же для `Table` / ячеек по правилам DSL.

Один контрол дизайн-системы в форме может получать пропсы **динамически** через **один** модуль — **configHook** (React-хук): это и настройка экземпляра Hexa, и место, где задаются колбэки (`onClick`, `onChange`, …) как часть возвращаемого объекта пропсов. Отдельного типа «обработчик события контрола» в DSL под это не вводится (см. пример `jsons/handlers/button.config-hook.ts`).

Действует принцип **«одна настройка — один способ»** ([tooling.md → Принцип](./tooling.md#принцип-одна-настройка--один-способ-задания)): каждый проп экземпляра — либо статическое поле в DSL, либо результат хука, но не оба сразу.

**Вход:** один аргумент — **formSlice**: настройки и стейт формы вместе (конфиг + значения полей и всё, что нужно для вывода пропсов). Тип компонента ДС задаётся самим модулем (один файл — один вид контрола), отдельный параметр не передаётся.

**Язык:** модули configHook — **только TypeScript** (`.ts`); загрузка в tool/viewer см. [tooling.md](./tooling.md) (раздел «Конфиг-хуки: только TypeScript»).

**Возвращаемое значение:** полный объект пропсов экземпляра компонента дизайн-системы **или `null`**. Хендлер — **единственный** источник пропсов Hexa для этого режима; статических полей тех же пропсов в DSL нет. Если хендлер вернул `null` — рендерер **не монтирует** компонент. Это единственный механизм управления видимостью: отдельного поля `visible` / `condition` в DSL нет.

**Реактивность:** хендлер — React-хук. Пересчёт пропсов обеспечивается штатными механизмами React и стейт-менеджера (подписки, контексты), без явного списка зависимостей в DSL.

**Два слоя контекста при колбэках:** при реализации функций внутри возвращаемых пропсов автор разделяет данные **инструмента/формы** (конфиг, `state`, идентификаторы контролов в DSL) и данные **слоя Hexa** (аргументы события как у нативного `onClick` / `onChange`, при необходимости актуальные пропсы экземпляра). Два блока явно разведены в сигнатурах внутренних функций или структуре аргументов — чтобы не смешивать KUNT и ДС. Это требование к **организации кода внутри** configHook, а не отдельный пункт плана.

## 2. Типизация конфига формы (генерируемый DSL)

Сейчас форма — это **JS-модуль** с `export default { ... }` ([tooling.md](./tooling.md)). Цель этапа — **строгая типизация** этого объекта относительно контракта DSL (`FormData` и связанные типы из `form-dsl.ts`): автодополнение и проверки при ручном редактировании файла формы, согласованность с тем, что ожидают tool и viewer.

Направления (на выбор или в комбинации, детали — при реализации): генерация рядом с формой **деклараций** (`.d.ts`), сохранение/экспорт из редактора варианта с **`satisfies`** / явной аннотацией корня, codegen при сохранении формы. Источник правды по форме полей — **типы DSL** в репозитории; паритет обоих приложений на одном контракте.

Связь с п.1: типы **`FormSlice`**, контролов (`ButtonControl`, …) и пропсов Hexa в configHook должны сходиться в одной цепочке, чтобы путь от `state[id]` до `ButtonProps` был проверяемым.

**Загрузка в transpile viewer (и при встраивании как библиотеки):** конфиг формы и модули **configHook** должны подключаться через **динамический `import()`** (или эквивалент с отдельными чанками), а не статическим бандлом всего каталога форм. Так сохраняются **чанкование**, **lazy loading** и предсказуемый размер начального бандла у потребителя движка.

## 3. Превью в редакторе

Превью формы работает на **той же** библиотеке Hexa UI, что и холст редактора. Не должно быть двух разных рендереров / двух библиотек — одна реализация для проектировщика и для превью.

## 4. Полная поддержка компонентов ДС

- **Все экспортируемые компоненты** из `@kaspersky/hexa-ui` (index) доступны в палитре. Список не хардкодится — формируется из экспортов пакета.
- **Все пропсы** каждого компонента доступны для настройки. Список свойств и подсказки типов берутся из TypeScript-описаний (`.d.ts`), а не из ручного перечня.

## 5. Storybook и прод

- **Normalization Tool** — выносится в **аддон (плагин) Storybook**: редактирование и просмотр форм из DSL внутри Storybook.
- **Transpile viewer** — становится **движком рендеринга** DSL: тот же движок подключается и к аддону Storybook (отрисовка форм в сторибуке), и к **продакшен-приложениям** (встраивание форм у пользователя без отдельной реализации рендера).
