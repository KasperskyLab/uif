## 3. Превью в редакторе

Одна библиотека **Hexa UI** для холста и превью — без расхождения рендереров.

**Открытые вопросы** (вернёмся при работе над этим пунктом роадмапа):

- ❓ **Grid и Table: `rows` / `cols` и configHook.** Сейчас в DSL явно задаются `rows` и `cols`, а на холсте normalization-tool **и сетка, и таблица** в основном строятся по этим полям; хук влияет на оформление контейнера (**`<Grid />`** / **`<Table />`**), но не на размерность. Не должны ли **рендер холста/превью** в редакторе в какой-то мере опираться и на возврат хука (в т.ч. на согласованные с DSL размеры), при том что **в артефакте формы** строки и колонки по-прежнему хранятся в DSL?

- ❓ **Table: тулбар и выбор строк.** Контрол `table` рендерится как Hexa **`<Table />`**, но тулбар сейчас **дублируется по смыслу**: в DSL остаётся **`toolbar`** (инспектор + **статический превью** над таблицей), а нативный тулбар Hexa можно задать только из **configHook** (`ITableProps['toolbar']`); на холсте хук таблицы не подмешивается. **Открыто:** свести рендеринг тулбара к **одному контракту** — перенести конфиг/рендер в **config hook** (и при необходимости упростить или переопределить роль поля в DSL), **плюс** добавить **рабочий пример** хука с **выбранными строками** (`rowSelection`, связка с состоянием формы / `FormSlice`, чтобы элементы тулбара реагировали на выделение).

## 4. Полная поддержка компонентов ДС

- **Перечень компонентов** и настройки из **всех** экспортов `@kaspersky/hexa-ui` и их **`.d.ts`**, без ручных списков.
- **Кастомный React-компонент** — подключение из **файла** (импорт модуля), с теми же принципами безопасной загрузки и паритетом **редактор / рендерер**, что у configHook (детали реализации — на этапе работы над пунктом).

## 5. Storybook и прод

**Normalization tool** — аддон Storybook. **Transpile viewer** — общий **движок рендера** DSL для Storybook и прод-приложений.

---

## 6. Архитектура кода и нейминг

- **Убрать дублирование:** каждый тип контрола обрабатывается **максимально общими** механизмами; всё **частное** выносится на **слой параметризации** (дескрипторы, конфиг маппинга, типы пропсов), а не копипаста веток в рендерерах и превью.
- **Нейминг (выровнять):** текущие **normalization tool** / **KUNT** → целевое имя **`kant-editor`**; **transpile viewer** → **`kant-renderer`** (пакеты, документация, UI-строки — по мере закрытия этапа).

---

## 7. Совместимость с JSON UI Builder и слой адаптеров

- **`form-dsl` и артефакты форм** — сделать **обратно совместимыми** с **JSON UI Builder**: один и тот же контракт данных (или явный мост версий / нормализация при импорте), без расхождения семантики контролов и корня формы.
- **Transpile viewer** — пересадить на **адаптеры** и **Auto UI**: вьюер не содержит монолитного маппинга «все типы подряд», а опирается на слой адаптеров к общему рантайму.
- **Auto UI** — **полная миграция на React** (отказ от устаревшего стека в пользу React end-to-end в рамках этой линии работ).

Детали контрактов, границы адаптеров и план миграции Auto UI — фиксировать по мере старта этапа (отдельные заметки или планы в [plans/](./plans/), по аналогии с п.1).

---

## 8. UI самих инструментов: только Hexa UI и, в перспективе, DSL + рендерер

- **Оба инструмента** (**normalization-tool** и **transpile-viewer**) для **собственного интерфейса** — оболочка, настройки, экраны, панели — должны опираться **только на компоненты Hexa UI** из **`@kaspersky/hexa-ui`**, без произвольной вёрстки «мимо» дизайн-системы там, где есть подходящий компонент ДС.
- **Целевое состояние:** инструменты **рисуют свой UI из DSL** и общего **рендерера** (тот же контракт «конфиг → дерево контролов → Hexa»), то есть **dogfooding**: редактор и просмотрщик не живут на отдельном ad-hoc UI-слое, а переиспользуют тот же движок, что и пользовательские формы (с оговорками по спец-режимам вроде холста/DnD — детали на этапе проектирования).
