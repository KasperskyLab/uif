# UI Normalization Tooling

**Назначение:** единая спецификация туллинга для **config-driven** UI-форм: описание формы — это **JS-объект (DSL)**, а не разметка и не ad-hoc React в вьюхах. Два приложения в монорепозитории закрывают разные этапы жизненного цикла этого конфига.

**Цели:** один формат данных формы; визуальное редактирование на этапе разработки; быстрый просмотр результата «как у пользователя» без редактора; рендер через **Hexa UI** как текущую реализацию.

**Не цели этого документа:** пошаговые CI-скрипты, точные версии зависимостей, полный справочник всех полей DSL (он живёт в типах в коде).

**Границы:** спека описывает роль пакетов, общую модель «конфиг → UI» и сценарии использования. Детали API отдельных хуков и компонентов — в `README.md` / `AGENTS.md` соответствующих пакетов.

**Оглавление раздела:** [README в этой папке](./README.md).

---

## Кратко

**Config-driven** означает: форма задаётся декларативным объектом (`name`, `id`, `schema`, `handlers`, `elements` и дерево контролов с типами вроде `button`, `input`, `grid`, `tabs` и т.д.). Вью не собирает форму руками из `useForm` + кучи компонентов — источник истины **файл `.ts`-модуля** с `export default { ... }`.

- **Tool** (`kaspersky-ui-normalization-tool`) — **dev-time**: выбор каталога на диске, WYSIWYG (палитра, холст, свойства), сохранение в **`.ts`**, предпросмотр, экспорт, шаблоны, обработчики и валидация в инспекторе.
- **Transpile viewer** (`kaspersky-ui-normalization-transpile-viewer`) — **только визуализация**: тот же каталог и те же **`.ts`**-формы, меню файлов, рендер DSL в UI на Hexa UI **без** полноценного редактора.

Оба приложения читают форму через **`parseFormTs`**: исходник транспилируется (**Sucrase**, как у `configHook`), затем выполняется динамический `import` blob-модуля.

## Границы продукта и паритет приложений

**Источники элементов в редакторе** — ровно две категории:

- **Компоненты дизайн-системы** — всё, что экспортируется из `@kaspersky/hexa-ui`.
- **Кастомные компоненты** — подключаются отдельно.

Третьих источников нет: фича, которой нужен обход ДС или ещё один класс источников, **вне скоупа** нормализации.

**Паритет KUNT и transpile viewer:** каждая возможность из [roadmap](./roadmap.md) доводится **сразу в обоих** пакетах. Фича считается закрытой только при согласованном DSL, семантике и поведении (без отставания редактора или просмотрщика).

## Модель данных (DSL)

Корень формы:

- **`name`**, **`id`** — заголовок и идентификатор.
- **`schema`** (опционально) — описание полей данных формы.
- **`handlers`** (опционально) — события уровня формы → пути к файлам-скриптам.
- **`elements`** — дерево контролов: семантические **`type`** (`button`, `text`, `input`, контейнеры `grid` / `row` / `table` / `tabs` и др.), **`id`**, плюс привязки к данным, валидация, условия видимости/доступности, обработчики на контроле.

Реализация на Hexa UI **не дублируется в DSL как отдельный метатип** — маппинг «семантика → компонент дизайн-системы» живёт в туле/вьюере (реестр дескрипторов / рендерер).

Подробные типы и сериализация: **`shared/types/form.ts`** — модель корня формы (`FormData`, `FieldSchema`); **`shared/normalization-form-dsl/form-dsl-core.ts`** — контролы и функции; **`form-dsl.ts`** — публичный entry; в пакетах **`src/types/form-dsl.ts`** реэкспортирует entry (alias `@normalization/form-dsl`).

<a id="normalization-one-setting-principle"></a>

## Принцип: одна настройка — один способ задания

Каждый параметр (проп) экземпляра компонента в форме имеет **ровно один** источник значения:

- **Статический** — задаётся в UI панели свойств, хранится как поле контрола в DSL. Значение фиксировано и не зависит от рантайма.
- **Динамический** — вычисляется хендлером настройки (хук, подключаемый модулем). В DSL для этого пропса **нет отдельного поля** — он целиком под контролем хука.

Совмещение обоих способов для одного пропса **запрещено**: нет merge, нет override статики динамикой. Если проп нужен динамически — он убирается из панели свойств и DSL-полей контрола.

<a id="normalization-config-hooks"></a>

## Конфиг-хуки экземпляров компонентов ДС (контракт)

В режиме **динамической** настройки один контрол дизайн-системы получает пропсы через **один** модуль — **configHook** (React-хук). Это и конфигурация экземпляра Hexa, и место для колбэков (`onClick`, `onChange`, …) **внутри** возвращаемого объекта пропсов. Отдельного типа «обработчик события контрола» в DSL под этот сценарий нет. Ориентир по стилю: пример `jsons/handlers/button.config-hook.ts` в пакете **normalization tool**.

Для режима configHook действует принцип **«одна настройка — один способ»** (см. раздел выше): пропсы Hexa целиком из хука, без дублирующих полей того же набора в DSL.

**Вход:** один аргумент — **`FormSlice`** (конфиг, стейт и всё необходимое для вычисления пропсов). Семантический тип контрола задаётся **самим модулем** (один файл — один вид), отдельный параметр не передаётся.

**Возвращаемое значение:** полный объект пропсов экземпляра компонента ДС **или `null`**. Хук в этом режиме — **единственный** источник этих пропсов. **`null`** — не монтировать компонент; отдельных полей `visible` / `condition` в DSL для этого нет.

**Реактивность:** хук выполняется как обычный React-хук; пересчёт — через React и стейт-менеджер, **без** явного списка зависимостей в DSL.

**Два слоя контекста в колбэках:** при реализации функций внутри возвращаемых пропсов автор явно разделяет контекст **формы/инструмента** (конфиг, `state`, идентификаторы контролов в DSL) и контекст **Hexa** (аргументы события, при необходимости актуальные пропсы экземпляра). Разведение — в сигнатурах или структуре аргументов вспомогательных функций, чтобы не смешивать KUNT и ДС.

### Язык модулей и загрузка в браузере

Модули **configHook** пишутся **только на TypeScript**: файл **`.ts`**, `export default` — функция `(formSlice) => пропсы | null`.

- В DSL и в пикере **KUNT** для `configHook` — **только** пути к `.ts`.
- **Normalization tool** и **transpile viewer** перед `import()` в браузере **транспилируют** TS → JS (`sucrase`), затем выполняют модуль из blob-URL.
- Обработчики в `handlers` у других контролов могут быть `.js` или `.ts` — ограничение **только** для configHook.
- Тип аргумента — **`FormSlice`** (`state` + `config.elements`), см. `form-dsl.ts`. Возврат: для кнопки — **`ButtonProps | null`**, для текста — **`TextProps | null`** (`@kaspersky/hexa-ui`).

<a id="normalization-dsl-typing-loading"></a>

## Типизация DSL и загрузка модулей (целевое состояние)

**Типизация:** форма — **TS-модуль** `export default { ... }` (типизация относительно **`FormData`** из **общего модуля DSL** в монорепозитории): автодополнение и проверки при правке, согласованность с tool и viewer. Направления (на выбор или вместе): `.d.ts` рядом с формой, вариант с **`satisfies`** / аннотацией при экспорте, codegen при сохранении. Источник правды — **один shared-исходный модуль**, который оба приложения **импортируют** (без отдельного publishable пакета под DSL). Закрытый этап **п.2.1** и отдельный **п.2.2** (типизация динамических `import()`): [plan-dsl-typing.md](./plans/plan-dsl-typing.md), [roadmap §2](./roadmap.md).

Связь с configHook: цепочка **`FormSlice`** → контролы (`ButtonControl`, `TextControl`, …) → пропсы Hexa в хуке должна быть **проверяемой** (например от `state[id]` до `ButtonProps` / `TextProps`).

**Динамическая загрузка:** при использовании **transpile viewer** как **библиотеки** (встраивание в Storybook, прод и т.д.) конфиг формы и модули **configHook** подключают через **динамический `import()`** (или эквивалент с отдельными чанками), а не одним статическим бандлом всего каталога форм — ради **чанкования**, **lazy loading** и предсказуемого размера начального бандла у потребителя движка.

## Сценарии

**Сценарий 1 — разработка формы**  
Открыть **Tool** → выбрать каталог → создать/открыть **`.ts`** → собрать форму на холсте → сохранить. При необходимости — предпросмотр и экспорт DSL.

**Сценарий 2 — проверка без редактора**  
Открыть **Transpile viewer** → выбрать тот же каталог → выбрать файл формы в дереве → убедиться, что конфиг корректно отображается в Hexa UI.

**Сценарий 3 — обмен артефактами**  
Каталог с **`.ts`-формами** и вспомогательными `handlers/*` передаётся как единица; tool и viewer ожидают **Chrome/Edge (desktop)** из‑за File System Access API.

```
  kaspersky-ui-normalization-tool/          # WYSIWYG-редактор (dev)

    src/utils/directoryHandleStorage.ts   # IndexedDB (тот же DB name, ключ viewer)

```

## Память выбранного каталога

После успешного выбора папки **`FileSystemDirectoryHandle`** сохраняется в **IndexedDB** (база `kaspersky-ui-normalization-fsa`, отдельные ключи: `normalization-tool` и `transpile-viewer`). При следующем открытии страницы приложение восстанавливает handle и запрашивает **`queryPermission` / `requestPermission`** для режима `read`; при `granted` диалог выбора каталога не показывается. Новый выбор каталога перезаписывает сохранённый handle.

У **normalization-tool** при **`window.__E2E_DEMO__`** персистентность отключена (Playwright).

### Параметр URL открытой формы

В адресной строке поддерживается query-параметр **`form`** — относительный путь к **`.ts`** файлу формы (как в дереве), например `?form=demo-form.ts` или `?form=sub%2Fform.ts`. При смене открытого файла URL обновляется (`history.replaceState`); после перезагрузки страницы, если каталог уже восстановлен из IndexedDB, форма из `form` открывается автоматически. У **normalization-tool** синхронизация с URL отключена в режиме E2E.
