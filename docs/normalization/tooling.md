# UI Normalization Tooling

**Назначение:** единая спецификация туллинга для **config-driven** UI-форм: описание формы — это **JS-объект (DSL)**, а не разметка и не ad-hoc React в вьюхах. Два приложения в монорепозитории закрывают разные этапы жизненного цикла этого конфига.

**Цели:** один формат данных формы; визуальное редактирование на этапе разработки; быстрый просмотр результата «как у пользователя» без редактора; рендер через **Hexa UI** как текущую реализацию.

**Не цели этого документа:** пошаговые CI-скрипты, точные версии зависимостей, полный справочник всех полей DSL (он живёт в типах в коде).

**Границы:** спека описывает роль пакетов, общую модель «конфиг → UI» и сценарии использования. Детали API отдельных хуков и компонентов — в `README.md` / `AGENTS.md` соответствующих пакетов.

**Оглавление раздела:** [README в этой папке](./README.md).

---

## Кратко

**Config-driven** означает: форма задаётся декларативным объектом (`name`, `id`, `schema`, `handlers`, `elements` и дерево контролов с типами вроде `button`, `input`, `grid`, `tabs` и т.д.). Вью не собирает форму руками из `useForm` + кучи компонентов — источник истины **файл `.js`-модуля** с `export default { ... }`.

- **Tool** (`kaspersky-ui-normalization-tool`) — **dev-time**: выбор каталога на диске, WYSIWYG (палитра, холст, свойства), сохранение в `.js`, предпросмотр, экспорт, шаблоны, обработчики и валидация в инспекторе.
- **Transpile viewer** (`kaspersky-ui-normalization-transpile-viewer`) — **только визуализация**: тот же каталог и те же `.js`-формы, меню файлов, рендер DSL в UI на Hexa UI **без** полноценного редактора.

Оба приложения читают один и тот же формат через **`parseFormJs`** (динамический импорт модуля из текста файла).

## Два компонента туллинга

| | **Normalization Tool** | **Transpile Viewer** |
|---|------------------------|----------------------|
| **Пакет** | `packages/kaspersky-ui-normalization-tool` | `packages/kaspersky-ui-normalization-transpile-viewer` |
| **Роль** | Построение и правка формы | Просмотр формы «как есть» |
| **Редактирование** | Да (холст, DnD, панель свойств, undo/redo) | Нет |
| **Файлы** | `.js` в выбранном каталоге (File System Access API) | То же |
| **Рендер** | Холст + режим предпросмотра (`FormPreview`) | `FormRenderer` по загруженному `FormData` |
| **Тесты** | Vitest + Playwright e2e | По необходимости (см. пакет) |

## Модель данных (DSL)

Корень формы:

- **`name`**, **`id`** — заголовок и идентификатор.
- **`schema`** (опционально) — описание полей данных формы.
- **`handlers`** (опционально) — события уровня формы → пути к файлам-скриптам.
- **`elements`** — дерево контролов: семантические **`type`** (`button`, `text`, `input`, контейнеры `grid` / `row` / `table` / `tabs` и др.), **`id`**, плюс привязки к данным, валидация, условия видимости/доступности, обработчики на контроле.

Реализация на Hexa UI **не дублируется в DSL как отдельный метатип** — маппинг «семантика → компонент дизайн-системы» живёт в туле/вьюере (реестр дескрипторов / рендерер).

Подробные типы и сериализация: `src/types/form-dsl.ts` в каждом из пакетов (копии/общая логика должны оставаться согласованными).

## Принцип: одна настройка — один способ задания

Каждый параметр (проп) экземпляра компонента в форме имеет **ровно один** источник значения:

- **Статический** — задаётся в UI панели свойств, хранится как поле контрола в DSL. Значение фиксировано и не зависит от рантайма.
- **Динамический** — вычисляется хендлером настройки (хук, подключаемый модулем). В DSL для этого пропса **нет отдельного поля** — он целиком под контролем хука.

Совмещение обоих способов для одного пропса **запрещено**: нет merge, нет override статики динамикой. Если проп нужен динамически — он убирается из панели свойств и DSL-полей контрола.

### Конфиг-хуки: только TypeScript

Модули **configHook** (хендлеры настройки экземпляра компонента ДС, п.2 роадмапа) пишутся **только на TypeScript**: файл с расширением **`.ts`**, `export default` — функция с сигнатурой `(formSlice) => пропсы | null`.

- В DSL и в пикере файлов в **KUNT** для configHook допускаются **только** пути к `.ts`.
- **Normalization tool** и **transpile viewer** перед динамическим `import()` в браузере **транспилируют** исходник TypeScript в JavaScript (зависимость `sucrase`), затем выполняют модуль из blob-URL.
- Обработчики событий у других контролов (пути в `handlers`, не configHook) по-прежнему могут быть `.js` или `.ts` — ограничение относится **только** к configHook.
- Аргумент хука — **`FormSlice`** (`state` + `config.elements`): тип экспортируется из `form-dsl.ts` в каждом пакете. Для кнопки возвращаемое значение типизируется как **`ButtonProps | null`** из `@kaspersky/hexa-ui` (сквозная типизация в рантайме tool и viewer).

## Сценарии

**Сценарий 1 — разработка формы**  
Открыть **Tool** → выбрать каталог → создать/открыть `.js` → собрать форму на холсте → сохранить. При необходимости — предпросмотр и экспорт DSL.

**Сценарий 2 — проверка без редактора**  
Открыть **Transpile viewer** → выбрать тот же каталог → выбрать файл формы в дереве → убедиться, что конфиг корректно отображается в Hexa UI.

**Сценарий 3 — обмен артефактами**  
Каталог с `.js`-формами и вспомогательными `handlers/*.js` передаётся как единица; tool и viewer ожидают **Chrome/Edge (desktop)** из‑за File System Access API.

```
    src/types/form-dsl.ts                 # DSL, parseFormJs, formToJs, …

    src/components/                      # холст, свойства, предпросмотр, экспорт
  kaspersky-ui-normalization-transpile-viewer/   # просмотр DSL
    src/utils/directoryHandleStorage.ts   # IndexedDB (тот же DB name, ключ viewer)
    src/utils/formUrlSync.ts              # ?form= — путь открытой формы
    src/types/form-dsl.ts                 # тот же контракт загрузки
    src/components/FormRenderer.tsx       # визуализация FormData
```

## Память выбранного каталога

После успешного выбора папки **`FileSystemDirectoryHandle`** сохраняется в **IndexedDB** (база `kaspersky-ui-normalization-fsa`, отдельные ключи: `normalization-tool` и `transpile-viewer`). При следующем открытии страницы приложение восстанавливает handle и запрашивает **`queryPermission` / `requestPermission`** для режима `read`; при `granted` диалог выбора каталога не показывается. Новый выбор каталога перезаписывает сохранённый handle.

У **normalization-tool** при **`window.__E2E_DEMO__`** персистентность отключена (Playwright).

### Параметр URL открытой формы

В адресной строке поддерживается query-параметр **`form`** — относительный путь к `.js` файлу формы (как в дереве), например `?form=demo-form.js` или `?form=sub%2Fform.js`. При смене открытого файла URL обновляется (`history.replaceState`); после перезагрузки страницы, если каталог уже восстановлен из IndexedDB, форма из `form` открывается автоматически. У **normalization-tool** синхронизация с URL отключена в режиме E2E.
